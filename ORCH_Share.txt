BUILD SPEC — Merchant API product-title A/B test
(TitleTest.gs is the reference implementation; the mechanics below match it.)

Footing: Merchant API v1, raw UrlFetchApp + ScriptApp.getOAuthToken() + content scope, script under itilemosaic@gmail.com. Same pattern as the diagnostics script (proven working this session).

No native title-experiment API exists — build it as a DIY cohort test:

1. Cohort split. Split the test SKUs into control vs variant. The tool holds the cohort map itself (offer_id -> control/variant, in a config tab). A custom_label (e.g. custom_label_4 = titletest_control / titletest_variant) is optional — only if you also want to filter the cohorts inside Google Ads / MC reporting; set it in the feed or via productInputs.insert into the supplemental source.

2. Baseline FIRST — before any title change. Capture per-SKU performance via accounts.reports.search on product_performance_view (impressions, clicks, conversions, conversion value) over a trailing window, keyed by offer_id. Write it to the Cesar-owned Sheet. This is the pre-change data point — it must happen before step 3.

3. Title swap. Write variant titles via accounts.productInputs.insert into a SUPPLEMENTAL data source, body field productAttributes.title. NOT productInputs.patch on the primary: the nightly feed overwrites a primary patch, and Google advises against mixing patch and supplemental for the same attribute. The supplemental source needs a merge rule that takes title FROM it — set in the Merchant Center UI ("Advanced data source management" add-on), standalone accounts only. Without that rule the primary feed's title keeps winning and the variant never serves. Control keeps its baseline title.

4. Measure. After the test window, re-pull product_performance_view for both cohorts and compare variant-vs-control deltas (difference-in-differences vs the baseline).

Baseline reassurance (for the "lose it permanently" concern): Merchant Center retains historical product performance queryable retroactively within its retention window — the pre-change point isn't gone if it isn't snapshotted this instant. Capture it now on today's footing anyway (no new setup); no reason to leave the clean point uncaptured.

Fit / guardrails — reason within these:

- Feed precedence (P1 — WooCommerce is source of truth for product data). The processed MC product is built from the primary data source (the catalog feed Sheet, fed from WC / USA Tile). A variant title written into the supplemental source only survives the nightly feed re-processing if the supplemental source has a merge rule that takes title from it — the default rule prefers the primary feed, so a naive supplemental write is ignored. Confirm variant titles persist against the feed before trusting the test (wait one feed cycle, then re-read the processed title). Coordinate so this doesn't fight the USA Tile master sync or the differentiated iTileMosaic layer.
- Review before live (P3). Variant titles are reviewed before they go live on the catalog.
- Measurement by design (P4). The baseline + post-window measurement plan is part of the build, not bolted on afterward.
- Rate / latency. Products update at most twice per day per product; there is a several-minute processing lag between a write and when the processed product reflects it — don't read back immediately.
